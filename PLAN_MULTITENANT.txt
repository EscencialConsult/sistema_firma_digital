CONTEXTO
--------
Escencial Consultora es el proveedor de la plataforma (SUPER_ADMIN).
Las empresas cliente son los tenants (cada una con su ORG_ADMIN y usuarios).
DIDIT maneja el KYC con un workflow separado por empresa.
Supabase es el backend (un solo proyecto, aislamiento por RLS).


ROLES DEL SISTEMA
-----------------
  SUPER_ADMIN   → Facundo / Escencial. Ve todo. Crea empresas. Gestiona planes.
  ORG_ADMIN     → El responsable de cada empresa cliente. Gestiona sus usuarios y contratos.
  USER          → Empleado/firmante de una empresa cliente.


ARQUITECTURA DECIDIDA
---------------------
- Un solo proyecto Supabase (no uno por tenant)
- Aislamiento por organization_id en todas las tablas via RLS
- organization_id y role inyectados en el JWT via custom_access_token_hook
- DIDIT: un workflow por empresa (workflow_id guardado en organizations table)
- Storage: estructura orgId/userId/verificationId/archivo
- Edge Functions leen organization_id del JWT, no hacen subquery a la DB

PASO 1.5 — custom_access_token_hook
--------------------------------------
  -- Esta función inyecta organization_id y role en el JWT en cada login.
  -- Es la pieza central del multi-tenant: evita subqueries en RLS.

  CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event JSONB)
  RETURNS JSONB
  LANGUAGE plpgsql STABLE
  AS $$
  DECLARE
    claims        JSONB;
    user_data     RECORD;
  BEGIN
    SELECT role, organization_id
    INTO user_data
    FROM public.users
    WHERE id = (event->>'user_id')::UUID;

    claims := event->'claims';
    claims := jsonb_set(claims, '{user_role}', to_jsonb(COALESCE(user_data.role, 'USER')));
    claims := jsonb_set(claims, '{organization_id}', to_jsonb(user_data.organization_id));

    RETURN jsonb_set(event, '{claims}', claims);
  END;
  $$;

  GRANT EXECUTE ON FUNCTION public.custom_access_token_hook TO supabase_auth_admin;

  -- Luego en Supabase Dashboard:
  -- Authentication → Hooks → activar "Custom Access Token"
  -- Seleccionar la función custom_access_token_hook

PASO 1.7 — Trigger para propagar organization_id automáticamente
-----------------------------------------------------------------
  -- Cuando se crea un documento, heredar el org del owner

  CREATE OR REPLACE FUNCTION public.fn_set_document_org()
  RETURNS TRIGGER LANGUAGE plpgsql AS $$
  BEGIN
    IF NEW.organization_id IS NULL AND NEW.owner_id IS NOT NULL THEN
      SELECT organization_id INTO NEW.organization_id
      FROM public.users WHERE id = NEW.owner_id;
    END IF;
    RETURN NEW;
  END;
  $$;

  CREATE TRIGGER trg_document_org
    BEFORE INSERT ON public.documents
    FOR EACH ROW EXECUTE FUNCTION public.fn_set_document_org();

  -- Mismo patrón para identity_verifications (desde user_id)

PASO 5.1 — Proceso de onboarding de nueva empresa
----------------------------------------------------
  1. Empresa firma contrato con Escencial
  2. SUPER_ADMIN entra a /super-admin/organizations/new
  3. Crea la organización en la plataforma
  4. Va a DIDIT Console → crea un nuevo workflow para esa empresa
     (sube el logo, configura colores, activa los pasos de verificación)
  5. Copia el Workflow ID y lo pega en la plataforma
  6. Crea el usuario ORG_ADMIN con email y contraseña temporal
  7. Le manda las credenciales al responsable de la empresa
  8. El ORG_ADMIN entra, cambia la contraseña y empieza a crear sus usuarios


PASO 5.2 — Proceso de baja de empresa
----------------------------------------
  1. SUPER_ADMIN marca is_active = false en la organización
  2. Los usuarios de esa org no pueden loguearse (RLS o middleware lo bloquea)
  3. Los datos se conservan por obligación legal (5 años según los T&C)

================================================================================
  DECISIONES TÉCNICAS TOMADAS
================================================================================

  1. Un solo proyecto Supabase (no uno por tenant)
     Razón: más simple, suficiente para el volumen esperado, migrable después.

  2. Aislamiento via RLS + JWT claims (no schemas separados)
     Razón: performance, mantenimiento simple, recomendado por Supabase.

  3. Una cuenta DIDIT, workflows separados por empresa
     Razón: confirmado por DIDIT. Escencial paga consolidado, cobra a sus clientes.

  4. Storage con orgId como carpeta raíz
     Razón: recomendado por Supabase. Permite RLS eficiente en storage.objects.

  5. Flujo de usuarios: ORG_ADMIN crea usuarios (no autoregistro)
     Razón: evita usuarios sin organización asignada. Más control.

  6. Rol SUPER_ADMIN separado de ORG_ADMIN
     Razón: Facundo necesita ver todo sin estar asignado a ninguna org específica.
